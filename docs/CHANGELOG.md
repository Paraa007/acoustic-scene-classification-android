# Changelog

Historische Entscheidungen und Implementierungs-Notizen für die Acoustic-Scene-Classification-App. Aus `CLAUDE.md` ausgelagert, damit der Hauptkontext schlank bleibt.

**2026-05-06 — Live-Screen-Polish 2: per-Second Emojis, Pause mit Timer-Picker, Wizard-Hardening gegen inkompatible Modell × Methoden-Combos**
- **Per-Second-Kreise zeigen Emoji.** Im AVG-Modus sind die zehn 28dp-Kreise unter der „Show Live Data"-Toggle-Zeile bisher nur Prozent-Kringel — bei zehn nebeneinander unleserlich, vor allem weil 1s-Klassen oft hin- und herspringen. Jeder Kreis sitzt jetzt in einer vertikalen Mini-Zelle: oben das Emoji der vorhergesagten Klasse (13sp, Zentriert), darunter der Kreis (28dp). Vor dem ersten Inferenz-Tick steht ein „·" als Platzhalter. Implementation in `LiveRecordingFragment.ensureCards`: aus jeder `ConfidenceCircleView` wird jetzt `LinearLayout(VERTICAL) { TextView(emoji); ConfidenceCircleView }`. `MethodSectionViews` bekommt zusätzlich eine `perSecondLabels: List<TextView>`, die im `render()`-Pfad parallel zu den Kreisen aktualisiert wird (`label.text = result?.sceneClass?.emoji ?: "·"`). `setConfidence(animate=false)` bleibt für das schnelle Befüllen.
- **Pause mit Timer-Picker.** Bisher war Pause ein einfacher Toggle ohne Frist — User wollten zurück, mussten erst aufstehen, kamen vom Café zurück und mussten manuell weiterklicken. Jetzt öffnet der Pause-Button auf der Live-Seite den existierenden `ModernDialogHelper.showPauseDurationDialog` (das Dialog-Layout `dialog_pause_duration` wird sowieso schon für den LongInterval-Picker mitbenutzt). Optionen: `No timer (manual resume)` · `5 min` · `10 min` · `30 min` · `1 h`. Bei einer Timer-Wahl wird `MainViewModel.pauseSession(autoResumeAfterMs)` mit dem Millis-Wert aufgerufen; der ViewModel feuert ein `viewModelScope.launch { delay(ms); resumeSession() }` und merkt sich die Deadline in `UiState.userPauseDeadlineElapsedMs` (bereits existentes Feld). „No timer" entspricht `pauseSession(null)` und verhält sich wie der alte Pause. Manuelles Resume funktioniert weiterhin und cancelt den Auto-Resume-Job.
- **Pause-Countdown im Status-Header.** Wenn ein Pause-Timer läuft, zeigt die Statuszeile oben (`liveStatusLabel`) jetzt `Paused · m:ss` (oder `h:mm:ss` bei langen Pausen) und tickt jede Sekunde herunter. Knackpunkt: Der `sessionTimerJob` im ViewModel emittiert während `isPaused` keine neuen UiStates, also würde die Zeile einfrieren. Lösung: ein fragmentseitiger `pauseCountdownJob`, der per `viewLifecycleOwner.lifecycleScope.launch` jede Sekunde `statusLabel.text = labelForState(viewModel.uiState.value)` aufruft, solange `isPaused && userPauseDeadlineElapsedMs != null`. Wird gestartet/gestoppt von `managePauseCountdown(state)` aus dem `render()`-Pfad und in `onDestroyView` defensiv gecancelt. `labelForState` selbst bekam einen frühen Branch: ist `isPaused` true, wird die Deadline ausgewertet und formatiert — der `AppState.Paused`-Branch (Interval-Aufnahme-Pause) bleibt für nicht-User-Pausen unangetastet.
- **Repair 1 — Wizard hängt nicht mehr nach Modell-Wechsel.** `MainViewModel.wizardSetModels()` filterte bisher nur die `intervalMethodsByModel`-Map auf die aktuell ausgewählten Modelle, seedete aber keine locked-default-Methode für neu hinzugefügte. Folge: User aktiviert in Step 1 ein zweites Modell, geht in Step 4 (Methoden), die Card des neuen Modells hat null Häkchen → `canAdvance()` returnt false (verlangt `intervalMethodsByModel[model].isNotEmpty()`), Weiter-Button bleibt grau, kein UI-Feedback warum. Fix: `wizardSetModels` ruft jetzt `models.associateWith { name -> kept[name] ?: setOf(ModelTrainingDuration.defaultSubMode(name)) }` und seedet damit garantiert immer mindestens den locked default. Zusätzlich werden inkompatible Methoden (siehe Repair 2) aus dem `kept`-Set rausgefiltert, falls jemand z.B. ein 10s-Modell durch ein 1s-Modell ersetzt während STANDARD geticked war.
- **Repair 2 — Wizard graut inkompatible Methoden aus.** Die Inferenz validierte nicht, ob ein gewähltes `LongSubMode` mit der Trainingsdauer eines Modells kompatibel ist. Konkret: `STANDARD` füttert die ganzen 10s ans Modell (passt nur zu `_10s_`-Modellen), `FAST` schickt einen 1s-Slice (passt nur zu `_1s_`-Modellen), `AVERAGE` zerteilt in 10×1s-Clips (passt nur zu `_1s_`-Modellen). Mismatch crashte entweder beim Tensor-Reshape oder lieferte Müll-Wahrscheinlichkeiten. Jetzt:
  - `LongSubMode` hat eine neue Property `sliceSeconds` (STANDARD=10, FAST=1, AVERAGE=1) und `isCompatibleWith(modelTrainingSeconds)`.
  - In `WizardFragment.renderIntervalMethods`: pro Modell wird via `ModelTrainingDuration.secondsForFilename(model)` die Trainingsdauer ermittelt; pro Checkbox dieses Modells wird `isCompatibleWith(duration)` geprüft. Inkompatible Boxen sind disabled, kriegen den Sublabel `not available (model trained on Xs clips)` (neuer String `wizard_method_incompatible`).
  - `MainViewModel.wizardToggleIntervalMethod` rejectet inkompatible Toggles silently (defense-in-depth, falls programmatisch ein `setOnCheckedChangeListener` doch durchrutscht).
- **Repair 2b — Mixed-Duration im Continuous geblockt.** Im Continuous läuft EIN Sub-Mode gegen ALLE selektierten Modelle. Mischt der User 10s + 1s-Modelle, ist keine Wahl korrekt. Fix in `WizardFragment.renderClipDuration`: `durations = state.selectedModels.map { ... }.toSet()`. Wenn `durations.size > 1`, erscheint ein Hinweis-Text oben (neuer String `wizard_clip_mixed_durations`), der erklärt warum nichts wählbar ist und auf „back oder Interval" verweist. Pro Sub-Mode wird gegen `durations.all { sub.isCompatibleWith(it) }` geprüft — passt nicht alles, wird die Card mit `alpha = 0.4f` und ohne Click-Listener gerendert (neue `enabled`-Param in `addPickRow`). Zusätzlich erweitert `WizardViewState.canAdvance()` die ClipDuration-Bedingung um genau diese Compat-Prüfung — damit der „Next"-Button auch hart auf „off" geht und nicht durch Stale-State-Schwung weiterspringt.
- **Build verifiziert** (`./gradlew assembleDebug` → BUILD SUCCESSFUL, nur preexistierende Deprecation-Warnings in `HistoryActivity.onBackPressed` und `VolumeLineChartView.scaledDensity`).

- **Ergebnis einer Grilling-Session, noch nicht implementiert.** Vollständige Spec in `docs/UI_REDESIGN_WIZARD.md`. Hier nur der Kurzabriss der getroffenen Entscheidungen, damit beim Implementieren nicht erneut diskutiert werden muss.

Bottom-Nav wird komplett entfernt — Navigation läuft nur noch über die Welcome-Page-Buttons + Back-Pfeile. `RecordingMode.devOnly` wird sinnlos (AVG immer verfügbar). Alle `isDevMode`-Verzweigungen fallen weg. Pref-Keys `long_sub_modes_user` werden obsolet, `long_sub_modes_dev` bleibt (umbenennen oder einfach so lassen). Alte History-Records mit `mode: "DEV"/"USER"` bleiben in SharedPrefs, das Feld wird nur nicht mehr gerendert (Mode-Badges werden durch ein Config-Label ersetzt: `🧠 model_xyz · Continuous · Standard 10s` o. ä.).
- **Wizard-Flow als neue Konfigurations-UI.** Variante C: 5 Schritte für Continuous, 6 Schritte für Interval, jeweils mit Übersichts-Seite als letzter Schritt. Continuous: Modelle → Continuous/Interval → Clip-Dauer → Session-Dauer → Übersicht. Interval: Modelle → Continuous/Interval → Pausen-Intervall → Methoden pro Modell → Session-Dauer → Übersicht. Jede Schritt-Seite hat einen Header-Text wie „Wähle ein oder mehrere Modelle." / „Wie soll die App aufnehmen?" / „Wie lang soll eine Aufnahme sein?" etc. Schrittweise Zurück-Navigation per Top-Bar-Pfeil + Hardware-Back. Auf der Übersichts-Seite ist jede Sektion klickbar = springt zum entsprechenden Schritt. Wizard-State liegt in `MainViewModel` als `WizardStep` Sealed Class.
- **Session-Dauer als neues Konzept** — gilt für beide Pfade. Gemeinsamer Picker: 5min · 15min · 30min · 1h · 3h · 6h · 12h · Manuell stoppen. „Manuell stoppen" = aktuelles Verhalten (läuft bis Stop). Bei Auto-Ende: clipgenauer Soft-Stop (laufenden Cycle inkl. Inferenz + History-Record zu Ende führen, dann stoppen, keine neue Iteration). Im Header der Live-UI als konzentrische Stopp-Uhr (äußerer Ring = Session, innerer Ring = Cycle, bei AVG in 10 Segmente unterteilt).
- **Multi-Model in beiden Pfaden** (vorher Interval-only). Im Continuous wird in der Recording-Loop pro Cycle über `inferenceByName` fan-out'd. Die Per-Model-Methodenwahl (Standard/Fast/Avg) bleibt Interval-only — im Continuous läuft pro Modell eine Methode (= globaler Continuous-Submode).
- **Live-UI: Bar-Distribution pro Modell × Methode statt Confidence-Kreis.** Der große Confidence-Kreis fliegt komplett raus. Pro gewähltem Modell eine eigene Card, vertikal gestapelt. Card-Header: 🧠 + Modell-Dateiname. Pro aktiver Methode eine Sub-Sektion (Header `1 — Standard` / `2 — Fast` / `3 — Avg`). Pro Sub-Sektion 9 horizontale Bars für die 9 Klassen, sortiert nach Wahrscheinlichkeit absteigend, jede Bar in der Klassenfarbe aus `SceneClass.color`. Emoji + Klassenname + Prozent in Standard-Textfarbe (NICHT in Klassenfarbe). Bei AVG zeigt die Bar-Distribution den laufenden Durchschnitt; pro Card ein „Show Live Data"-Toggle (default aus), der aufgeklappt 10 kleine `ConfidenceCircleView`-Kreise (bestehende Klasse) zeigt, die sich live einer pro Sekunde füllen.
- **Was fliegt aus dem Live-UI:** großer Kreis, Triangle-Layout, Top-Predictions-Card, Last-Predictions-Card, globaler Per-Second-Toggle, Mode-Auswahl-Buttons, Mode-Timeline, Modell-Edit-Stift, LongSubMode-Checkboxen, Multi-Model-Container, Continuous/Interval-Buttons. Volume-Graph bleibt **permanent sichtbar** unten (kein Toggle mehr — Power-Impact ist vernachlässigbar, der Audio-Path läuft eh).
- **Pause/Resume jetzt in beiden Modi** (vorher Interval-only). Clipgenaue Pause-Semantik wie der Soft-Stop: laufender Cycle wird zu Ende geführt (inkl. Inferenz + History-Record), erst dann pausiert die Schleife. Resume startet einen frischen Cycle. Während Pause hält der Session-Timer an. Pausen werden in der CSV als eigene synthetische Records geschrieben (`mode_label = "PAUSE"`, alle Klassen-Spalten leer, neue Spalte `pause_duration_sec`) — so kann man in Pandas/Excel direkt `.filter(mode_label == "PAUSE")`. In der History-UI erscheinen Pause-Records als graue Trennlinie zwischen Aufnahmen, nicht als eigene Kachel.
- **Volume in der CSV (neu).** Zwei neue Spalten: `volume_mean` und `volume_peak`, jeweils 0.0 – 1.0 mit 3 Nachkommastellen. Mean = Mittelwert aller RMS-Samples über die Aufnahmedauer (~30 Hz × Cycle-Dauer). Peak = lautester einzelner RMS-Wert. Bei Multi-Model: Audio identisch für alle Modelle in einem Cycle, also Volume nur einmal pro Cycle (nicht pro Modell-Inferenz). Bei AVG-Records zusätzlich pro 1s-Clip mitgeführt — `perSecondClips`-Format wird zu `1:Park:79%:mean=0.421:peak=0.583|2:Park:85%:mean=0.398:peak=0.502|…`. `PredictionRecord` bekommt zwei neue Felder, `volumeFlow` muss pro Cycle aggregiert werden (Sammeln im RecorderCallback, Aggregat-Berechnung beim Cycle-Ende).
- **Results-Summary-Screen (neu).** Nach Stop oder Auto-Stop landet der User auf einer neuen Seite, die für jedes Modell × Methode die finalen Bar-Distributions auf einen Blick zeigt (aggregiert über alle Cycles der Session). Pro Card: Anzahl Cycles, häufigste Klasse über die Session, Durchschnitts-Volume. Buttons `Zurück zur Home Page` und `Zur History` (öffnet History mit dieser Session vorausgewählt).
- **Kein Reconfigure mid-Session.** Während eine Session läuft, gibt's nur Stop. Nach Stop → Results Summary → Zurück zur Home Page → entweder „Letzte Config nutzen" mit kleinen Änderungen oder „Neue Session starten" (Wizard ab Schritt 1). Klare Hartline, keine Fork-Semantik.
- **Evaluation-Feature bleibt Interval-only**, aber Timing folgt jetzt dem im Wizard gewählten Pausen-Intervall (vorher hardcodiert 30 min). Bei Pausen-Intervall = 1 h kommt die Evaluation-Notification jede Stunde. Im Continuous explizit keine Evaluation-Card (würde alle 10 s erscheinen, unbedienbar).

Der Lookup `inferenceByName` mappt Modellname → `ModelInference`-Instanz (Single-Model = `mapOf(primary to modelInference)`; Multi-Model = `_allInOneModelNames.zip(allInOneInferences).toMap()`). Letzteres wird bei jedem `updateLongSubModeUi`-Aufruf idempotent angestoßen — sorgt dafür, dass das Map nach Multi-Model-Picker-Änderungen immer genau die aktiven Modelle enthält und Defaults gemerged sind.

Vorteil: Modelle, die kurz aus dem Multi-Model-Set entfernt und wieder hinzugefügt werden, behalten ihre Methodenauswahl.

Sobald gewählt, steht da `every <interval-label> · <locked-default-method>` mit der pro-Modell passenden Methode (Standard für 10s-Modelle, Fast für 1s-Modelle).
- CSV-Format: die existierenden 3 Spalten `long_standard`/`long_fast`/`long_average` bleiben strukturell gleich, der **Inhalt** ist jetzt pipe-serialisiert wenn mehrere Modelle aktiv: `"m1:Park:80%|m2:Park:75%"`. Format pro Eintrag: `"<modelName>:<class>:<XX>%"` für neue Records, `"<class>:<XX>%"` ohne Präfix für Records mit `modelName == null` (legacy). Konsistent mit dem bestehenden `per_second_clips`-Pattern (`"1:Natur:79%|2:Natur:85%|…"`). Schema bleibt stabil — neue/entfernte Modelle ändern keine Spalten.
- History-Detail-Dialog: die „Method comparison"-Sektion zeigt jetzt **Modell als Top-Header** (`🧠 modellname`) und darunter eingerückt eine Distribution-Stack-Sektion pro `(Modell, Sub-Mode)`-Kombination. Bei Records ohne `modelName` (vor dem Migrationszeitpunkt) wird auf `record.modelName` gefallen. Reine Doppelschleife `for model { for sub { renderStack() } }` über die existierende `createDistributionItemWithProgress`-Render-Funktion — kein neues UI-Pattern, gleicher NestedScrollView mit `maxHeight=600dp`.
